# Offline Artist Knowledge

Nova Music Lab should not scrape biographies or copy large copyrighted pages into the app bundle.

The safe architecture is:

1. Use open structured sources first.
2. Store compact metadata offline.
3. Generate original emotional/biographical prose inside the app from facts, tags and archive evidence.

## Recommended Sources

- MusicBrainz: artist IDs, aliases, countries, begin dates, tags and release groups.
- Wikidata: public identifiers, country, occupation, inception dates and structured facts.
- Curated public links: compact manual corrections when automatic matching points at the wrong artist.
- Cover Art Archive: cover-art metadata by MusicBrainz release ID.
- ListenBrainz dumps: optional large listening-behavior data for recommendation research, not needed for basic artist bios.

## Why Not Download Every Artist Into The Frontend?

The full music universe is too large for a browser bundle. A complete MusicBrainz or Wikidata mirror belongs in a local/server-side database, not inside `src/data`.

The app should ship with a compact generated cache for the current archive. Later, the uploader can enrich a new user's artists by:

1. reading their uploaded history,
2. extracting unique artists,
3. matching those artists to the local compact cache,
4. optionally running the enrichment script for missing artists.

## Generated File

`src/data/offline_artist_knowledge.json` is generated by:

```bash
npm run knowledge:artists
```

For a fast smoke test:

```bash
npm run knowledge:artists -- --limit=5
```

The script respects MusicBrainz rate limits and writes a compact JSON file that the emotional engine can read offline.
Wikidata is fetched in SPARQL batches and cached under `scripts/.cache/offline_artist_knowledge`, so repeated runs do not keep hitting public services.

## Current Scope

This pass builds a local offline knowledge cache for the 100 artists already present in the bundled archive. It stores:

- archive rank, plays, genre, country, top tracks and top albums,
- MusicBrainz artist ID, aliases, origin fields, activity dates, ISNI values and tags,
- Wikidata QID, public description, inception/birth dates, official website URLs and Wikimedia image URLs when available,
- Wikidata members, occupations and instruments when structured statements are available,
- curated correction facts for ambiguous artists where an automatic match is weaker than public evidence,
- MusicBrainz album release groups with first release dates,
- compact emotional seed text that the engine can score without new network calls.

The emotional engine reads this file through `src/utils/offlineArtistKnowledge.ts`, so artist, album and song dossiers can use real structured context while still writing original bilingual prose.
The Data Quality center also reads the same file to build an enrichment queue: a ranked list of artists needing profile work, Wikidata facts, descriptions, member/role facts, official links, public images or dated album context.

## World-Scale Next Phase

For "all artists in the world", use a separate local or server-side database instead of the frontend bundle:

1. Download official MusicBrainz database dumps for artist, alias, area, tag and release-group metadata.
2. Optionally download Wikidata JSON dumps, then extract only music-artist entities and public identifiers.
3. Build a local SQLite or DuckDB index with normalized artist names, aliases, countries, tags, active years and release groups.
4. During user upload, extract unique artist names, match them against the local index, and write a small per-user cache into the app.
5. Keep biographies as generated original summaries from structured facts, not copied text from websites.

This gives Nova Music Lab the feeling of a giant offline artist brain without shipping gigabytes of data to every browser.
